# ADR-002: Резидентность из VoltDB, не из MySQL

**Дата:** 2026-04-30
**Статус:** Принято

## Контекст

В заголовках ЭСФ-документов уже есть колонка `is_seller_non_resident`
(`tinyint`, NULL допускается). На первый взгляд, мы могли бы взять её
напрямую и не подключать VoltDB.

Однако в начале работы проекта мы провели разведку и обнаружили, что:

- значения колонки заполнены **непоследовательно**;
- для многих документов стоит `NULL`;
- для одного и того же `seller_tin` в разных документах могут быть
  разные значения этого флага;
- источник этого флага — пользовательский ввод при заполнении ЭСФ,
  а не справочник.

В то же время в корпоративном **справочнике налогоплательщиков**
(VoltDB) есть актуальные данные по всем налогоплательщикам с признаком
резидентности (`resident`), который синхронизирован с реестром.

## Рассмотренные варианты

### A. Использовать `is_seller_non_resident` из MySQL

**Плюсы:**

- Никаких новых зависимостей;
- Один источник данных;
- Простой SQL-запрос.

**Минусы:**

- **Ненадёжный источник** (заполняется вручную пользователями);
- Множественные значения для одного BIN;
- NULL в большом проценте записей;
- Не обновляется при смене статуса налогоплательщика.

### B. Использовать справочник налогоплательщиков из VoltDB (выбрано)

**Плюсы:**

- **Авторитетный источник** — синхронизация с реестром;
- Одно значение на BIN;
- Покрытие ~78% всех BIN'ов нашего графа;
- Дополнительные атрибуты (наименование, статус, тип).

**Минусы:**

- Новая зависимость (voltdbclient);
- Разный формат BIN (12-значный VARCHAR vs MySQL bigint);
- VoltDB-специфика (лимиты IN-clause, отсутствие GROUP BY 1);
- Сетевая латентность на 100K lookup'ов.

### C. Свой справочник в JSON / parquet

**Плюсы:**

- Нет внешней системы в runtime;
- Дёшево читать.

**Минусы:**

- Где взять полный справочник? Нужна ETL-пайплайн из VoltDB;
- Устаревает между обновлениями;
- Лишний шаг синхронизации.

### D. Гибрид: VoltDB как primary, fallback на MySQL

**Плюсы:**

- На случай недоступности VoltDB — деградирует, но работает.

**Минусы:**

- Удвоенная сложность кода;
- Несогласованность: для одного BIN VoltDB говорит «резидент», MySQL — «нерезидент». Кому верить?
- Не нужно для прототипа.

## Решение

Выбран **справочник налогоплательщиков из VoltDB** как **единственный**
источник резидентности.

Колонка `is_seller_non_resident` в исходных ЭСФ-документах **не
используется** ни в одном из расчётных скриптов.

Реализовали:

- `voltdb_client.py` — низкоуровневый адаптер;
- `volt_resolver.py` — пакетный lookup с кэшем и нормализацией BIN;
- интеграция в `kz_index.py::enrich_with_voltdb()`.

## Последствия

### Положительные

1. Каждый узел графа получает корректную метку `is_non_resident`;
2. Дополнительно получаем `volt_name` — это **сильно улучшает читаемость**
   отчётов для бизнес-аудитории (теперь видим «ТОО Асем-2», а не голый BIN);
3. Возможность в будущем добавить фильтрацию по статусу/типу
   (например, исключить налогоплательщиков в статусе «закрыт»).

### Отрицательные

1. Зависимость от **двух баз** в runtime — оба должны быть доступны;
2. ~25 секунд из 80 общего времени уходит на VoltDB lookup;
3. Для ~21.5% узлов BIN не находится — приходится трактовать оптимистично.

### Что упростилось / усложнилось дальше

| Сценарий | Стало |
|---|---|
| Расширить набор атрибутов узлов (тип, статус, регион) | **Проще** — VoltDB уже подключён |
| Развёртывание в air-gapped окружении | **Сложнее** — нужны оба сервера |
| Валидация качества данных | **Проще** — есть авторитет для cross-check |
| Реал-тайм запросы | Не цель прототипа, но VoltDB поддерживает |

## Граничный случай: BIN не в справочнике

При получении ответа из VoltDB могут возникнуть три ситуации:

| Ситуация | Что делает движок |
|---|---|
| BIN найден, `resident = 1` | `is_non_resident = False` (резидент) |
| BIN найден, `resident = 0` | `is_non_resident = True` (нерезидент) |
| BIN найден, `resident = NULL` | `is_non_resident = None` |
| BIN не найден | `is_non_resident = None` |

В обоих случаях `None` узел трактуется как **оптимистичный резидент**
при расчёте `kz` (если он source — `kz=1`, иначе считается по обычной
формуле). Это сознательный размен, см. [Ограничения](../../docs/limitations.md).

## Что мы НЕ делаем (намеренно)

- Не кэшируем VoltDB-снимок локально (parquet/db). Всегда читаем свежий.
- Не используем статус активности налогоплательщика — пока неясно, как
  трактовать «приостановленных» в кз-индексе. Это задел на будущее.
- Не используем тип (юрлицо/ИП/физлицо) — все типы участвуют в графе
  одинаково.
