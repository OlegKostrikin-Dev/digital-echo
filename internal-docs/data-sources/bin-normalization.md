# Нормализация BIN

Самый коварный технический нюанс проекта — **разные форматы идентификатора (БИН/ИИН) в MySQL
и VoltDB**. Если на этом ошибиться, движок будет молча возвращать
неверные результаты, потому что lookup в VoltDB будет промахиваться.

## Проблема в одну строку

| Источник | Формат идентификатора |
|---|---|
| MySQL — заголовки ЭСФ (`seller_tin`/`customer_tin`) | `bigint unsigned`, **без ведущих нулей** |
| VoltDB — справочник налогоплательщиков (колонка `tin`) | `VARCHAR(12)`, **с ведущими нулями** |

Один и тот же БИН выглядит по-разному:

```text
В MySQL:  40000537
В VoltDB: 000040000537
```

При прямой склейке `WHERE tin IN ('40000537')` в VoltDB найдётся 0 строк,
хотя налогоплательщик существует.

## Формат БИН/ИИН РК

БИН (юрлица) и ИИН (физлица) РК — **12-значные числа**:

| Позиции | Значение |
|---|---|
| 1–6 | Дата (YYMMDD для физлиц / YYMMxx для юрлиц) |
| 7 | Тип (`4` или `5` — юрлицо, `0`–`9` для физлиц) |
| 8–11 | Порядковый номер |
| 12 | Контрольный разряд |

Поскольку первые 6 позиций — это дата, **они часто начинаются с нуля**
(например, январь 2004 года → `040128`). Поэтому у БИН/ИИН **обязательно**
есть ведущие нули в первой половине числа.

Когда идентификатор сохраняется как `bigint`, эти нули **отбрасываются**.

## Как мы решаем

Файл `volt_resolver.py`:

```python
BIN_LENGTH = 12

def _pad(bin_val: str | int) -> str:
    """Нормализация BIN к 12-символьному виду с ведущими нулями."""
    s = str(bin_val).strip()
    return s.zfill(BIN_LENGTH) if s else s
```

Применяется **во всех точках** взаимодействия с VoltDB:

```python
unique = sorted({_pad(t) for t in bins if str(t).strip()})
```

## Защита от SQL-инъекции

При построении IN-clause мы дополнительно проверяем, что BIN — это только
цифры и ровно 12 символов:

```python
sanitized = [t for t in bins if t.isdigit() and len(t) == BIN_LENGTH]
```

Это убирает и потенциальные инъекции, и просто кривые входные данные.

## Загрузка из MySQL: явный dtype

Когда мы читаем `pd.read_sql(EDGES_QUERY, ...)`, без явного указания
типа pandas инферит `int64` для колонок с БИН (потому что в MySQL это
`bigint unsigned`). Это безвозвратно теряет ведущие нули **до того**, как
мы получим возможность их восстановить.

Решение — явный `dtype` параметр:

```python title="kz_index.py"
BIN_DTYPE = {"source": "string", "target": "string"}

with engine.connect() as conn:
    df_edges = pd.read_sql(EDGES_QUERY, conn, params=params, dtype=BIN_DTYPE)
```

После этого все идентификаторы — это python-строки, и `_pad()` корректно дошьёт
им нули перед запросом в VoltDB.

## Проверка корректности

Чтобы убедиться, что нормализация работает, в `voltdb_explore.py` есть
diagnostic-режим:

```bash
docker compose run --rm echo-engine python voltdb_explore.py
```

Он показывает, какой процент BIN'ов из графа находится в VoltDB.
Если процент сильно ниже 70% — это первый признак, что padding сломался.

## Граничные случаи

| Вход | Что происходит |
|---|---|
| `"180640000680"` (12 символов) | Без изменений |
| `"40000537"` (8 символов) | → `"000040000537"` |
| `40000537` (int) | Сначала `str()`, потом padding → `"000040000537"` |
| `""` или `None` | Возвращается пустая строка → отфильтруется |
| `"123ABC"` | Нормализуется, но потом `isdigit()` отсеет в `_fetch_chunk` |

## Что делать, если BIN не находится

Бывают две ситуации:

| Ситуация | Что делает движок |
|---|---|
| BIN не найден в справочнике налогоплательщиков | Помечается `is_non_resident = None`, считается резидентом «оптимистично» |
| BIN найден, но `resident = NULL` | То же — `is_non_resident = None` |

Это сознательный размен: лучше слегка завысить kz-индекс, чем штрафовать
компанию за пробелы в справочнике.

## Альтернатива: хранить БИН в MySQL как VARCHAR

Это **правильное решение** на долгую дистанцию: в строках ЭСФ-документов
есть колонка `consignee_tin VARCHAR(12)`, которая хранит номер правильно.
Но обе колонки `seller_tin` и `customer_tin` остались `bigint unsigned`
из исторических соображений.

Когда (если) источник ЭСФ перейдёт на `VARCHAR(12)` для всех полей с БИН/ИИН,
функция `_pad()` станет no-op, но останется как страховка.
