# Ограничения прототипа

Эта страница — честный список того, что **движок не умеет** или умеет
с оговорками. Цель — задать корректные ожидания для бизнеса и команды
разработки.

## Полнота источника данных

### Что не попадает в граф

| Источник стоимости | Учтён? | Почему нет |
|---|---|---|
| B2B-транзакции через ЭСФ | :material-check: | Ядро движка |
| B2C-розница | :material-close: | Не оформляется через ЭСФ |
| Трансферы между филиалами | :material-close: | Внутренние перемещения, не сделки |
| Импорт через таможню без ЭСФ | :material-close: | Параллельный канал |
| Государственные субсидии и трансферы | :material-close: | Не B2B-сделка |
| Услуги через зарубежные платёжки | :material-close: | Часто не имеют ЭСФ |
| Бартерные сделки | :material-close: | Регистрируются нерегулярно |

**Следствие:** наш индекс — это «индекс КС в ЭСФ-документированной части
B2B-оборота», а не «индекс КС всей экономики».

### Качество полей идентификатора (БИН/ИИН)

- Колонки `seller_tin` и `customer_tin` в исходных ЭСФ —
  `bigint unsigned`, **теряют ведущие нули**. Решено через явный dtype
  и нормализацию (см. [Нормализация BIN](data-sources/bin-normalization.md)).
- Около **21.5%** BIN'ов нашего графа **не находятся** в справочнике
  налогоплательщиков. Они трактуются как резиденты «оптимистично».

### Тестовые данные

В тестовой среде присутствуют:

- BIN'ы вида `123456789021` (ТОО «Асем-2»);
- ИП «Тестовый» (`920806450284`);
- BIN'ы с именами «Company NNN», «User NNN».

Они **не отфильтровываются** автоматически — нет надёжного маркера в данных.
На production нужно будет либо подключить чёрный список, либо договариваться
с источником о пометке тестовых записей (см. [Roadmap](roadmap.md)).

## Алгоритмические допущения

### «Всё или ничего» для нерезидентов

Все продажи нерезидента считаются 100% импортом, даже если он
теоретически мог купить часть товара у казахстанского производителя
и перепродать. На реальных данных это редкая ситуация, но возможная.

### «Оптимизм» для unknown

Если узел — источник (нет поставщиков) и его нет в справочнике
налогоплательщиков, он считается резидентом с `kz = 1.0`. Альтернатива — считать
«unknown» и исключать из агрегатов — даст более «честные», но
менее интерпретируемые результаты.

### Узлы внутри одного SCC получают одинаковый kz

Это математически корректно (см. [ADR-003](decisions/003-iterative-fixed-point.md)),
но в отчётах для бизнеса нужно объяснять. Аудитория может ожидать,
что у разных компаний цикла будут разные индексы.

### Не различаются типы товара

Индекс рассчитывается по **деньгам**, не по физическим единицам или
группам товаров. Если компания одновременно продаёт казахстанский хлеб
и импортные мобильники, индекс будет средневзвешенным. Для разделения
по группам нужны коды ТРУ, которых нет в текущем срезе.

### Период «жёсткий»

Расчёт идёт строго по `[date_from, date_to)`. Если у компании в этом
окне нет ни одной сделки, она просто не попадает в граф.
Транзакция, начавшаяся до окна и завершившаяся внутри, попадёт целиком —
потому что мы фильтруем по `turnover_date`.

## Производительность

### Текущие пределы

| Метрика | Лимит при 1 ГБ RAM |
|---|---|
| Узлы | ~500K |
| Рёбра | ~2M |
| Время прогона на 100K узлов | ~80 секунд |
| Время прогона на 500K узлов | ~10 минут (экстраполяция) |

### Узкие места

1. **VoltDB lookup** — 25 секунд на 100K BIN'ов;
2. **Fixed-point итерации** — 50 секунд на 244 итерации;
3. **MySQL запрос** — 5 секунд для 90 дней.

При расширении до года ожидается:

- MySQL ~30 секунд;
- VoltDB ~60 секунд;
- Итерации ~200 секунд.

## Отсутствующие фичи

### Фильтр тестовых данных

Нет автоматического исключения тестовых BIN'ов. Если на демо
аудитория попросит «без тестов» — нужно вручную перечислять BIN'ы.

### Web-интерфейс

Нет. Только CLI с консольным выводом. Web-морда планируется
(см. [Roadmap](roadmap.md)).

### Сравнение периодов

Нет встроенной возможности «сравнить kz за Q1 и Q2». Нужно запускать
два прогона и сравнивать вручную.

### Экспорт результатов

Нет автоматического сохранения в CSV/parquet. Можно перенаправить
stdout, но это текст, не структура.

### Аудит результатов

Нет журнала, какие параметры были использованы для конкретного отчёта.
Если кто-то спросит «а на каких данных был получен этот результат» —
ответить можно только примерно.

### Метрики качества

Нет встроенной валидации: «не выглядит ли результат подозрительно?»
(например, kz < 0 или > 1 не должны встречаться, но никто не проверяет
явно после `compute_kz_content`).

## Ограничения VoltDB integration

- Нет retry на временные сбои сети;
- Нет таймаутов на отдельный запрос (только глобальный сетевой);
- Нет circuit breaker — если VoltDB недоступен, движок падает.

Это приемлемо для batch-режима, но не для интерактивного use-case.

## Ограничения MySQL integration

- Нет connection pooling beyond `pool_pre_ping`;
- Запросы идут в один поток;
- Нет защиты от случайной выгрузки слишком большого периода
  (`--days 3650` спокойно запустится и положит сервер).

## Что следует сказать аудитории

> «Это **прототип** для проверки гипотезы. Он показывает, что подход
> работает и какого порядка результаты ожидать. Перед production
> нужны: расширение данных, фильтр тестов, web-интерфейс, отчётный
> архив. Дорожная карта — на странице Roadmap.»
