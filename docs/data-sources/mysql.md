# MySQL: источник B2B-транзакций

Данные электронных счетов-фактур (ЭСФ): **заголовки документов** и **строки
позиций** в корпоративном хранилище на базе MySQL (движок InnoDB). Из них
строятся рёбра графа «продавец → покупатель» и суммы сделок.

## Масштаб хранилища и срез по периоду

**Оценки порядка величин** (по снимку данных, на котором делался разбор и
документация):

| Что считаем | Порядок величины | Пояснение |
|---|---|---|
| Строк в таблице заголовков ЭСФ | **~1,3 млн** | один ряд ≈ один учётный документ (шапка) |
| Строк в таблице позиций (строк ЭСФ) | **~1,3 млн** | детализация сумм по документам; на один документ может приходиться несколько строк или одна |

Это **размер таблиц в хранилище целиком**, а не размер одного аналитического
среза.

**Срез для расчёта индекса** строится по **дате оборота** в строках ЭСФ
(`turnover_date`): в расчёт попадают только документы, чьи строки попадают в
выбранный календарный интервал (например, последние 90 дней). Поэтому в работу
попадает **доля** записей из полного объёма — см. ниже типовый порядок для
90-дневного окна.

## Таблицы и их роли

В документации ниже — **логические** имена ролей. Соответствие реальным именам
таблиц в вашей инсталляции смотрите в коде загрузки данных
(`services/backend/src/core/edges.py`).

| Роль в документации | Содержимое | Порядок размера (в хранилище) |
|---|---|---|
| `<headers_table>` | Шапки ЭСФ | ~1,3 млн записей |
| `<lines_table>` | Строки позиций ЭСФ | ~1,3 млн записей |

## Поля заголовка ЭСФ

| Колонка (роль) | Тип | Зачем |
|---|---|---|
| `invoice_id` | `bigint unsigned` (PK) | Связь шапки со строками |
| `seller_tin` | `bigint unsigned` | БИН/ИИН продавца — исток ребра |
| `customer_tin` | `bigint unsigned` | БИН/ИИН покупателя — приёмник ребра |

Поля, которые **не используем** как источник истины:

| Колонка | Почему |
|---|---|
| `is_seller_non_resident` | Заполнено непоследовательно; резидентность берём из справочника (VoltDB). См. [ADR-002](../decisions/002-voltdb-residency.md). |
| `total_price_with_tax` в шапке | Для веса рёбер берём сумму **без НДС** из строк |

## Поля строк ЭСФ

| Колонка | Тип | Зачем |
|---|---|---|
| `invoice_id` | `bigint unsigned` | Соединение со шапкой |
| `total_price_without_tax` | `decimal(17,2)` | **Вес ребра** (сумма по паре продавец–покупатель после агрегации) |
| `turnover_date` | `datetime` | **Отбор среза по периоду** |

!!! tip "Почему без НДС"
    НДС переуплачивается по цепочке; сумма с НДС дала бы **двойной счёт** на
    каждом шаге. Сумма без НДС ближе к добавленной стоимости и **аддитивна** по
    цепочке.

## Индексы (общее)

В продуктивной БД обычно есть составные индексы по парам продавец–покупатель–дата
и по идентификатору документа — это нужно для укладывания тяжёлого агрегирующего
запроса в приемлемое время. Детали имён — в комментариях в `edges.py`.

## Как из таблиц получаются рёбра (логика запроса)

```sql
-- Шаблон: имена таблиц подставляются из конфигурации кода.
SELECT
    h.seller_tin                    AS source,
    h.customer_tin                  AS target,
    SUM(d.total_price_without_tax)  AS weight,
    COUNT(DISTINCT h.invoice_id)    AS invoice_count
FROM <headers_table>           AS h
INNER JOIN <lines_table>       AS d
        ON d.invoice_id = h.invoice_id
WHERE d.turnover_date >= :date_from
  AND d.turnover_date <  :date_to
  AND h.seller_tin   IS NOT NULL
  AND h.customer_tin IS NOT NULL
  AND h.seller_tin <> ''
  AND h.customer_tin <> ''
GROUP BY h.seller_tin, h.customer_tin
```

## Производительность: один срез (пример 90 дней)

Типовой порядок на **тестовой** выборке при окне **90 дней** (именно столько
документов/строк **проходит фильтр по дате**, не весь миллион):

| Метрика | Порядок значения |
|---|---|
| Заголовков ЭСФ в срезе | ~100 тыс. |
| Строк позиций в срезе | ~100 тыс. |
| Уникальных рёбер (пар продавец–покупатель) после агрегации | ~106 тыс. |
| Время одного прогона основного запроса | порядка **5 с** |

На **365 днях** ожидаемо порядка **400 тыс.** рёбер и десятки секунд на запрос —
зависит от железа и нагрузки на БД.

## Профиль одной компании

Отдельные запросы с фильтром по `seller_tin` (продажи) и по `customer_tin`
(закупки), с группировкой по контрагенту — для карточки компании и конусов.

## Качество данных (наблюдения EDA, тот же срез что выше)

| Наблюдение | Масштаб | Действие |
|---|---|---|
| Отрицательные суммы (возвраты) | ~6 рёбер | Учитываем только `weight > 0` |
| Нулевые суммы | ~467 рёбер | Учитываем только `weight > 0` |
| Петли «сам на себя» | единичные | Не отбрасываем явно |
| Тестовые БИН | несколько | Не фильтруем; см. [Ограничения](../limitations.md) |
| Очень крупные суммы на ребре | до ~11,8 млрд ₸ | допустимо для холдинговых цепочек |
