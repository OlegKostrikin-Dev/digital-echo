# Архитектурные решения (ADR)

Architecture Decision Records — короткие документы, описывающие
**почему** мы выбрали то или иное решение и **что отвергли**.

Формат позаимствован у [Майкла Найгарда](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions).

## Когда добавлять новый ADR

- Решение влияет на структуру проекта (новая БД, новый язык, новый сервис);
- Между альтернативами есть нетривиальный размен;
- Будущему разработчику будет неочевидно, **почему** мы выбрали именно
  этот вариант.

## Шаблон ADR

```markdown
# ADR-NNN: Краткое название

**Дата:** YYYY-MM-DD
**Статус:** Предложено / Принято / Отклонено / Заменено на ADR-XXX

## Контекст

Что было известно на момент решения. Какая проблема стояла.

## Рассмотренные варианты

- Вариант A — плюсы / минусы
- Вариант B — плюсы / минусы
- Вариант C — плюсы / минусы

## Решение

Какой вариант выбран и почему.

## Последствия

Что изменилось в проекте. Какие новые ограничения.
Какие следующие решения станут проще / сложнее.
```
