# ADR-001: NetworkX как in-memory движок графа

**Дата:** 2026-04-30
**Статус:** Принято

## Контекст

Нам нужен движок графа для:

- хранения ~100K-200K узлов и ~100K-300K рёбер с атрибутами;
- BFS вверх и вниз;
- расчёта SCC и конденсации;
- итераций fixed-point по узлам.

В перспективе годовой выборки граф вырастет до ~500K узлов и
~1M рёбер. Десятилетней — ~5M узлов.

## Рассмотренные варианты

### A. NetworkX (выбрано)

**Плюсы:**

- Pure Python, ставится одной строкой `pip install networkx`;
- Хорошая документация и большое community;
- Из коробки SCC, конденсация, BFS, descendants/ancestors;
- Атрибуты на узлах и рёбрах прямо в API;
- Легко экспериментировать — REPL-friendly.

**Минусы:**

- Однопоточный, нет SIMD-оптимизаций;
- Хранение через dict-of-dict — оверхед памяти ~1KB на ребро;
- На 10M+ рёбер становится больно.

### B. graph-tool

**Плюсы:**

- C++ ядро, в 50–100 раз быстрее NetworkX на крупных графах;
- Эффективное хранение через scipy.sparse;
- Богатые алгоритмы.

**Минусы:**

- Сложная установка (нужен boost, нативные либы);
- Менее интуитивный API (нужно явно работать с PropertyMap);
- Pip-пакета нет — только conda или сборка из исходников;
- Для прототипа на 100K узлов это overkill.

### C. igraph (Python wrapper)

**Плюсы:**

- C-ядро, быстрее NetworkX;
- Поставляется в pip.

**Минусы:**

- API менее удобен (узлы — целочисленные индексы, не объекты);
- Меньше алгоритмов SCC/конденсации в готовом виде;
- Атрибуты узлов работают неуклюже.

### D. Кастомный код на scipy.sparse

**Плюсы:**

- Максимально быстрая итерация fixed-point через matrix multiplication;
- Полный контроль над памятью;
- Прямая совместимость с future-GPU реализациями.

**Минусы:**

- Высокая стоимость разработки и тестирования;
- BFS и SCC придётся писать самому или брать `scipy.sparse.csgraph`;
- Для прототипа избыточно.

### E. Neo4j / другая graph database

**Плюсы:**

- Персистентность, transactional, query language;
- Запросы вроде Cypher могут быть лаконичны.

**Минусы:**

- Внешняя система, ещё одно «движущееся» в архитектуре;
- Стоимость передачи данных туда-обратно велика;
- Cypher не подходит для итеративного fixed-point;
- Лицензионные нюансы у Neo4j Enterprise.

## Решение

Выбран **NetworkX** на этапе прототипа.

Аргументы за:

1. Скорость разработки важнее скорости исполнения. 80 секунд на полный
   прогон — приемлемо для batch-обработки.
2. API — самый быстрый способ итерировать гипотезы.
3. Установка — `pip install networkx==3.4.2`, работает «из коробки».
4. Когда (если) граф вырастет до 5M+ узлов — переход на graph-tool или
   scipy.sparse будет относительно прост, потому что мы в любом случае
   используем матричную модель в голове.

## Последствия

- Зависимость от Python и `networkx==3.4.2` в `requirements.txt`;
- Прогон требует ~1 ГБ RAM на 100K узлов (запас есть);
- Невозможна параллельная обработка нескольких независимых подграфов
  (можно было бы при scipy.sparse + multiprocessing);
- При переходе на production-объёмы понадобится бенчмарк и, вероятно,
  миграция на graph-tool. Кстати, [проверьте Roadmap](../roadmap.md).

## Выходы из решения

Если NetworkX «не вытянет», переход в порядке возрастающей сложности:

1. **graph-tool** — переписать `compute_kz_content` на нём, остальное оставить;
2. **scipy.sparse + кастомный код** — полный отказ от NetworkX, ручная BFS/SCC;
3. **Spark GraphFrames** — если объём перевалит за 100M+ узлов.
